# 広告制作ワークフロー（ad-workflow）

スライドショー／カルーセル型広告を、企画から実装・レビュー・提出まで迷わず進めるための段階的ガイド。

## このスキルの基本原則

- **いきなり実装しない。** まず企画から始める。
- **悩みと強みは1つに絞る。** 複数あっても主役は1つ。
- **AIの役割分担を守る:** 企画整理は ChatGPT（あるいはこのスキル内）、実装は Claude Code、レビューは Codex。
- **1スライド1メッセージ。** コピーは短く、CTAは最後に最も目立たせる。

## 使い方

ユーザーがこのスキルを呼び出したら、まず現在地を確認する。「これから新しい広告を企画しますか？それとも途中の段階ですか？」と短く聞き、以下のフェーズのうちどこから始めるかを決める。

すでに材料が揃っている場合は、フェーズをスキップして直接テンプレ生成に進んでよい。

---

## フェーズ 0: 現在地の確認

ユーザーに以下のいずれかを聞く（1回で済むよう、必要なものだけ）:

- 何の広告か（サービス／会社／架空 or 実在）
- 制作物の形式（スライドショー／カルーセル、表示比率、枚数、提出形式）
- 今どの段階か（企画ゼロ／企画あり実装前／実装済みレビュー前／レビュー済み修正中）

これがないと先に進めない情報は、**質問は最大4個まで**に絞って一度に聞く。曖昧でも進められる場合は仮置きで先に進める。

---

## フェーズ 1: レギュレーション確認

ユーザーが提出するレギュレーション（課題要項、PowerPoint指定など）があれば必ず最初に確認する。

確認項目:

- 広告の種類（スライドショー／カルーセル／静止画）
- 表示比率（16:9 / 1:1 / 9:16 / その他）
- 使用技術（HTML/CSS/JS、PowerPoint など）
- スライド枚数
- CTAの有無
- ロゴの有無
- 自動再生／ナビゲーションの有無
- 提出形式
- 禁止事項

レギュレーションと矛盾する企画は早めに止める。

---

## フェーズ 2: サービス設計

以下を順に確定する。ユーザーから引き出す質問は1回で4個まで。

| 項目 | 確認内容 |
|------|----------|
| サービス名 | サービスの名前 |
| サービス種別 | AI英会話 / 学習 / 観光 / EC / イベント など |
| 架空 or 実在 | どちらか |
| ターゲット | 年齢・職業・状況まで具体化 |
| ターゲットの悩み | **1つに絞る** |
| サービスの強み | **1つに絞る**（他では解決できない独自性） |
| 一番伝えたいこと | 1文で |
| メインコピー | 短く、最初に目に入る言葉 |
| タグライン | 補助コピー |
| CTA | 行動を明確にする言葉 |
| 避けたい表現 | 煽り／断定／誇大／幼すぎ／怪しさ等 |

不安・コンプレックス煽り、根拠のない断定、過度なAI表現、ターゲットと合わない幼すぎる表現は避ける。

---

## フェーズ 3: コンセプト固め

以下の型に当てはめて1〜2文に圧縮する:

```
[ターゲット]でも、[サービスの強み]によって、[悩み]を気にせず始められる。
[サービス名]は、[体験価値]を提供する[サービス種別]。
```

ここで違和感があれば、ターゲットか強みか悩みのどれかがズレている。フェーズ2に戻る。

---

## フェーズ 4: スライド構成

基本は4枚構成。

1. **悩み共感** — ターゲットの悩みを短く提示。商品説明はしない。
2. **解決提示** — サービスでどう悩みが軽くなるかを見せる。メインコピーを最も大きく。
3. **サービス価値・機能紹介** — 具体的な機能を3〜4個カードで。盛り込みすぎない。
4. **CTA** — サービス名、タグライン、CTAボタンを明確に。次の行動を迷わせない。

各スライドについて以下を埋める:

- 目的
- 表示コピー
- 補足コピー
- ビジュアル案
- 注意点

**応用:** 3枚構成では「悩み共感＋解決提示」を1枚に統合。5枚構成では「使い方ステップ」または「実績・声」を追加。

---

## フェーズ 5: デザイン方針確定

以下をユーザーと確定する:

- 全体の方向性: 信頼感 / 爽やか / ポップ / 高級感 / 近未来 / かわいい / シンプル / 力強い
- 配色: メイン / サブ / アクセント / 背景
- UIの雰囲気: Apple風ミニマル / カード型 / 雑誌風 / LP風 / 高級感重視
- ビジュアル方針: イラスト / 実写 / 3D / UI中心 / 抽象表現中心

**合わない組み合わせの代表例:**

- 社会人向け × 子どもっぽいイラスト
- 信頼感重視 × ネオン系の過度なAI表現
- 初心者向け × 強すぎる不安煽り
- 白ベース × 目立たないCTA

これらに該当しそうな場合は警告する。

## フェーズ 6: Claude Code 用 実装プロンプト生成

フェーズ2〜5で埋めた内容を以下のテンプレに差し込んで出力する。出力は**コードブロック**で囲み、ユーザーがそのままコピーできる状態にする。

```markdown

## 目的

HTML、CSS、JavaScriptを使って、[サービス名]のスライドショー／カルーセル型広告を作成してください。

[課題・制作条件・PowerPointレギュレーション]に沿って、[表示比率]のHTML表示内で成立する広告にしてください。既存ファイルがある場合は内容を確認し、必要に応じて修正または作り直してください。

ファイル作成、フォルダ作成、構造設計はClaude Code側で判断して構いません。

最終的に、ブラウザでHTMLを開くだけで広告が表示される静的サイトとして完成させてください。

## 今回の制作条件

広告対象：[サービス名]
サービス種別：[サービス種別]
表示形式：スライドショー／カルーセル型広告
表示比率：[16:9 / 1:1 / 9:16 / その他]
使用技術：HTML / CSS / JavaScript
スライド枚数：[3 / 4 / 5 枚]
実装方式：ビルド不要の静的HTML
確認方法：ブラウザでHTMLを直接開く

## 重要な前提

- React、Vue、Svelte などのフレームワークは使わない
- HTML / CSS / JavaScript のみ、ビルド環境なしで動作
- [表示比率]の表示領域で広告として成立させる
- スライドショー／カルーセル型広告にする
- [スライド枚数]枚構成
- 自動再生アニメーション、ドットナビゲーション、できれば前後ボタンを入れる
- CTAボタン、ロゴまたはサービス名（左上）を入れる
- 広告全体を中央配置
- 仮文字・テスト文言・仮デザインは残さない
- コンソールエラーが出ないようにする

## サービス情報

サービス名：[サービス名]
サービス種別：[サービス種別]
ターゲット：[ターゲット]
ターゲットの悩み：
- [悩み1]
- [悩み2]
- [悩み3]
サービスの強み：
- [強み1]
- [強み2]
- [強み3]
広告で一番伝えたいこと：[メインメッセージ]
メインコピー：[メインコピー]
タグライン：[タグライン]
CTA：[CTA文言]
最終コンセプト：[誰に向けて、どんな悩みを、どんな強みで解決するサービスなのかを1〜2文]

## デザイン方針

全体の方向性：[全体方向性]
色：
- メインカラー：[メインカラー]
- サブカラー：[サブカラー]
- アクセントカラー：[アクセントカラー]
- 背景色：[背景色]
UIの雰囲気：[UI雰囲気]
ビジュアル方針：[ビジュアル方針]
避けたい表現：
- [避けたい表現1]
- [避けたい表現2]
- [避けたい表現3]

画像・素材方針：
- 外部画像を無理に取得しない
- 著作権的に安全なCSS図形、SVG、抽象イラスト、UIモックを使ってよい
- 必要に応じて assets フォルダを作成してよい
- 画像生成が必要な場合は prompts.md にまとめる
- 本番素材がない場合でも、仮素材に見えすぎない品質で作る

## スライド構成

### Slide 1：悩み共感
目的：[悩みを提示し自分ごと化]
表示コピー：[コピー]
補足コピー：[補足]
ビジュアル案：[ビジュアル]
注意点：商品説明をしない、煽りすぎない

### Slide 2：解決提示

### Slide 3：サービス価値・機能紹介

表示コピー：[コピー]

- [要素1]

- 画面比率は[表示比率]を守る
- 広告本体は中央配置、ロゴ／サービス名は左上
- メインコピーは大きく最初に目に入る位置
- CTAは最終スライドで最も目立たせる
- ドットナビゲーションは下部中央
- 1スライド1メッセージ、文字を詰め込みすぎない、余白を十分に
- スクリーンショットで提出しやすい見た目に

## アニメーション要件

- 自動再生、切り替え間隔は4〜5秒、なめらかに
- 過度に派手な演出は避ける
- ドット／前後ボタンの操作で該当スライドへ
- 手動操作後も自然に自動再生へ戻る
- 可能ならホバー一時停止／タブ非アクティブ停止／prefers-reduced-motion 対応

## 実装要件

- ファイル分離してよいが保守性を優先（構成はClaude Codeが判断）
- どのHTMLを開けばよいか README.md に明記
- CSS変数で配色・余白を管理
- 仮テキスト・不要コメントは残さない
- コンソールエラーなし、相対パスで動く

## 推奨ファイル構成

      js/carousel.js

- ロゴ／サービス名が左上にある
- デザイン方針・ターゲットに合っている
- 仮文言なし、HTML/CSS/JS エラーなし
- ブラウザで直接開いて確認できる、README に開き方記載

## 仕上げ時の出力

1. 作成・修正したファイル一覧
2. どのHTMLを開けば確認できるか
3. 実装したカルーセル機能の概要
4. デザイン上の意図
5. レギュレーションに対して満たした点
6. まだ弱い点
7. Codexでレビューした方がよい点
```

## フェーズ 7: Codex 用 レビュー依頼テンプレ生成

Claude Code で実装が終わったら、以下を出力する。同様にコードブロックで囲む。

```markdown
# Codex レビュー依頼：スライドショー／カルーセル型広告

## 目的

Claude Codeで実装した [サービス名] のスライドショー／カルーセル型広告をレビューしてください。
コードの正しさ、広告としての伝わりやすさ、UIの見やすさ、ターゲットとの相性、提出物としての完成度を確認してください。
忖度なし、問題点は重要度順に。

## 前提情報

広告対象：[サービス名]
サービス種別：[サービス種別]
ターゲット：[ターゲット]
ターゲットの悩み：[悩み]
サービスの強み：[強み]
一番伝えたいこと：[メッセージ]
コンセプト：[1〜2文]

## 必須コピー

- サービス名：[サービス名]
- メインコピー：[メインコピー]
- タグライン：[タグライン]
- CTA：[CTA文言]

## デザイン方針

全体方向性：[方向性]
配色：メイン/サブ/アクセント/背景：[色]
UIの雰囲気：[雰囲気]
ビジュアル方針：[方針]
避けたい表現：[表現]

## 制作条件

表示形式：スライドショー／カルーセル型広告
表示比率：[表示比率]
スライド枚数：[枚数]
使用技術：HTML / CSS / JavaScript
ファイル構成：[実装後の構成]

## レビュー観点

1. **レギュレーション適合**: 表示比率・枚数・CTA・ロゴ位置・中央配置・仮文言なし・ブラウザで開ける
2. **広告としての完成度**: 1枚目で興味/共感、2枚目で解決、3枚目で機能、最終でCTA。1スライド1メッセージ、ターゲットに合った言葉、煽り/誇大表現なし
3. **スライド構成**: 流れがあるか、役割重複なし、情報量過多なし
4. **UI・デザイン**: 配色・余白・行間・文字サイズ・強弱・CTA視認性・ナビ見やすさ・トーン統一・スクショ耐性
5. **HTML**: 構造・見出し階層・要素選択・alt/aria・パス
6. **CSS**: 比率崩れ・レスポンシブ・CSS変数・重複・z-index・アニメ過剰・hover/focus
7. **JavaScript**: 自動再生/ドット/前後/キーボード/ホバー停止/タブ非アクティブ/復帰/状態管理/イベント増殖/コンソールエラー
8. **アクセシビリティ**: aria-label/aria-current/aria-live/キーボード/focus/コントラスト/prefers-reduced-motion/代替テキスト
9. **パフォーマンス・保守性**: 重い処理・外部依存・構成のわかりやすさ・README

## 出力形式

1. 総合評価（点数/100、一言評価）
2. 良い点（3〜5個）
3. 問題点（重要度 高/中/低 別、各「問題点」「理由」「修正方針」）
4. すぐ直すべき修正
5. できれば直す改善
6. コード修正案（HTML/CSS/JS の具体的修正案）
7. 広告コピー改善案（現状／問題点／改善案3つ）
8. デザイン改善案（配色／余白／CTA／ビジュアル）
9. 最終判断（このまま提出 / 軽微な修正後 / 重要な修正後 / 提出しない方がよい）

## 特に厳しく見てほしい点

- [厳しく見てほしい点1]
- [厳しく見てほしい点2]
- [厳しく見てほしい点3]

## 注意

忖度なしでレビュー。見た目がそれっぽくても、広告として弱い点があれば指摘。
学校課題として必要十分な範囲は考慮し、過剰な作り込みより提出物として効果的に見えることを優先。
修正案は実装者がそのまま直せるよう具体的に。
```

## フェーズ 8: 提出前チェック

ユーザーが Codex レビュー後の修正を終えたら、以下を1つずつ確認する:

- HTMLを直接開いて動く
- 指定比率が守られている
- スライドが自動再生される
- ドットナビゲーションが動く
- CTAが見える
- ロゴまたはサービス名がある
- 仮文字が残っていない
- コンソールエラーがない
- スクショで見ても広告として成立する
- ターゲットに合ったデザイン
- コピーが長すぎない
- 避けたい表現が入っていない
- README に開き方が書かれている
- Codex レビュー後の修正が反映されている

すべてOKなら提出。NGがあれば該当フェーズに戻る。

---

## 振る舞いのルール

- **ユーザーへの質問は1回で4個まで。** 多い場合は分けて聞く。
- **テンプレ生成時は必ず Markdown のコードブロックで囲む。** ユーザーがそのままコピーできる形にする。
- **埋まっていないプレースホルダ `[...]` を残さない。** すべて埋めてから出力。埋められない項目は確認する。
- **広告として弱い設計（ターゲットと方向性のミスマッチ等）に気づいたら指摘する。** 黙って通さない。
- **実装そのものはこのスキルでは行わない。** 実装は Claude Code、レビューは Codex に渡すための準備に徹する。
